# QueueStorm Investigator Test Plan

This plan is built from `SUST_Hackathon_Preli_Problem_Statement.pdf`, `SUST_Preli_Sample_Cases.json`, and the companion rubric/manual. It is intentionally a black-box API stress set for `POST /analyze-ticket`, not a unit-test plan for one implementation.

Comparison policy: the enum/core fields should match exactly. The prose fields are acceptable exemplars; a real service can use different wording if it is concise, operationally equivalent, and safe. `customer_reply` and `recommended_next_action` must never ask for PIN, OTP, password, full card number, or promise an unauthorized refund/reversal/unblock.

Coverage shape:

- 15 straightforward calibration cases at most.
- Evidence traps: explicit IDs, status contradictions, ambiguous histories, duplicate patterns, established-recipient patterns, missing relevant IDs.
- Normalization traps: Bangla, Banglish, Bengali numerals, mixed notation, phone formatting, shorthand amounts.
- Safety traps: prompt injection, refund promises, third-party contact, credentials included in the complaint, metadata override attempts.
- Routing/escalation traps: contested refunds, high-value cases, cash-out gaps not present in taxonomy, merchant/agent/channel conflicts.

Each JSONL line is one test case with `test_json` as the request body and `expected_response` as a valid response body.
